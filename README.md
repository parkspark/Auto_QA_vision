# 🎮 비전 기반 게임 QA 자동화 | 캐릭터 탐지 및 추적

**블랙박스 환경 기반의 엔드 유저 체감형 게임 QA 자동화**  
화면(픽셀)만 보고 플레이어블 캐릭터를 실시간으로 탐지·추적하는 “AI 테스터”

게임 내부 코드·메모리에 전혀 접근하지 않고, **오직 화면(픽셀)만 보고** 캐릭터(`character`)와 닉네임(`user_id`)을 탐지한 뒤, 닉네임 바로 아래의 캐릭터를 “내 캐릭터”로 선정(A안 후처리)하여 **동영상에서 실시간으로 추적**합니다. 연습 대상은 2D 픽셀 액션 게임 **던전앤파이터(Dungeon & Fighter)** 입니다.

> **Tech Stack**  
> `YOLO11s` · `ByteTrack` · `Computer Vision` · `Black-Box QA` · `Active Learning` · `PyTorch` · `CUDA` · `Flask` · `Streamlit`

---

## 📊 주요 지표
- **탐지 클래스:** 2종 (character · user_id)
- **현역 모델 정확도:** mAP50 0.931 (v3 · imgsz 1280)
- **학습 데이터:** 스틸 907 + 합성 7,070 (+ 영상 프레임 721)
- **ID 스위치:** 276 → 88 (hysteresis 락 ↓68%)

---

## 1. 프로젝트 개요
게임 클라이언트 내부의 소스 코드나 메모리 데이터에 전혀 접근하지 않는 **완전한 블랙박스(Black-Box) 환경**에서, 실제 플레이어의 시각(화면) 정보만으로 게임 플레이를 검증하는 자동화 테스트 시스템입니다. 화려한 이펙트와 UI가 산재한 인게임 화면 속에서 YOLO11s 모델이 **모든 플레이어 캐릭터(character)**와 **닉네임(user_id)**을 분리 탐지하고, “내 캐릭터”는 학습이 아니라 **후처리 정책**(가장 신뢰도 높은 닉네임 박스 바로 아래의 캐릭터 선택)으로 결정합니다. — 내 캐릭터와 파티원은 *시각적으로 구분 불가*하기 때문입니다.

> 🎯 **2단계 목표:** ① 정적 스크린샷 탐지 (v1~v2) → ② **동영상에서 캐릭터 실시간 추적** (v3~, 현재 단계)

## 2. 프로젝트 목적
- 🎯 **엔드 유저 관점의 검증:** 내부 API·로그로는 확인하기 힘든, 실제 모니터에 출력되는 렌더링 결과물 기반의 결함(캐릭터 낌, 이펙트 가림, 비정상 좌표 이동 등)을 플레이어 눈높이에서 자동 확인합니다.
- 🛡️ **비간섭적(Non-Intrusive) 테스트:** 게임 엔진·클라이언트에 영향을 주지 않고 영상 스트림만 분석합니다. 라이브 빌드에도 즉시 적용 가능한 QA 파이프라인입니다.
- ♻️ **반복 테스트 피로도 해소:** 수동으로 하던 장시간 플레이 모니터링·버그 재현을 비전 모델이 대체하여 회귀 테스트의 효율과 정확도를 극대화합니다.

## 3. 핵심 가치 & 기술적 도전
- 🔍 **다양한 상황 속 인식률 최적화:** 수천 가지 아바타 조합·스킬 이펙트·반투명 UI가 산재한 2D 픽셀 노이즈 속에서도 대상을 분리. 닉네임(원본 ~70×24px)을 잡기 위해 imgsz=1280 로 학습 해상도를 끌어올렸습니다.
- 🔒 **추적 정책의 진동 제어:** 진짜 병목은 탐지가 아니라 ‘내 캐릭터 선정 정책’의 진동이었습니다. hysteresis 락(coast-grace·challenge-frames)으로 ID 스위치를 276→88 로 줄였습니다.
- 📊 **데이터 기반 품질 지표:** 라벨 없는 영상에서도 추적률·ID 스위치·coast 프레임을 산출해 특정 맵·상황의 탐지 실패를 정량 진단합니다.

> 💡 **실측 인사이트:** 영상 도메인 적응(v3)으로 동영상 캐릭터 탐지율은 0.885 → **0.888** 로 올랐지만, pseudo-label 의 “가려진 파티원 미탐지(FN)·모션블러”로 정적 Recall 은 소폭 하락했습니다. ‘큰 데이터가 항상 낫다’가 아니라 도메인 트레이드오프를 **정량적으로** 드러냅니다.

## 4. 시스템 아키텍처 & 파이프라인
**STEP 1 라벨 변환** ➜ **STEP 2 합성 증강** ➜ **STEP 3 영상 프레임 병합** ➜ **STEP 4 모델 학습** ➜ **STEP 5 추적·평가**

- **STEP 1 라벨 변환:** labelme JSON → YOLO 포맷 · train/val seed 42 분할
- **STEP 2 합성 증강:** 직업별 스프라이트 7,070장으로 합성 프레임 생성
- **STEP 3 영상 프레임 병합:** 동영상 0.5fps 추출 → pseudo-label → train 병합
- **STEP 4 모델 학습:** YOLO11s · imgsz 1280 · 버전별(v1~v4) 학습
- **STEP 5 추적·평가:** ByteTrack + hysteresis 락 추적 · 추적지표 산출

> **능동학습 루프:** 학습된 모델이 새 스틸/영상 프레임을 1차 라벨링(pseudo-label)하고, 사람은 *처음부터 그리는 대신 O/X/? 검수만* 합니다. NPC·소환수 오탐 107건을 제거(v4 데이터셋)하는 식으로 데이터가 쌓일수록 라벨링 비용은 줄고 모델은 강해지는 선순환을 구현했습니다.

### 📂 파이프라인 스크립트 구성 (`scripts/`)
| 단계 | 스크립트 | 역할 |
|------|----------|------|
| 라벨 변환·분할 | `convert_labelme.py` | labelme JSON → YOLO 라벨 + train/val 분할(seed 42) |
| 합성 증강 | `synthesize.py` | 직업별 스프라이트 합성으로 학습 프레임 2,000장 생성 |
| 영상 프레임 추출 | `extract_frames.py` | 동영상 0.5fps 추출 + v2/v3 pseudo-label(conf 0.5 게이팅) |
| 영상 프레임 병합 | `merge_video_frames.py` | 영상 프레임을 train 셋에 병합 (val 오염 금지) |
| 모델 학습 | `train.py` / `resume_train.py` | YOLO11s · imgsz 1280 학습 · 중단 복구(resume) |
| 영상 추적·평가 | `track_video.py` | ByteTrack + hysteresis 락 추적, 추적지표 JSON 산출 |
| 라벨 검수 | `make_review.py` · `apply_review.py` | 클릭 O/X/? 검수 페이지 생성 → 원본 라벨 반영(백업 먼저) |

## 5. 탐지 & 추적 쇼케이스
- **영상 추적 데모:** ByteTrack과 hysteresis 락(안정적인 ‘내 캐릭터’ 락온)을 적용하여 안정적인 추적. (ID 스위치 대폭 감소)
- **버전별 탐지 비교:** v2 vs v3 (v3는 영상 프레임 721장을 추가해 동영상 도메인에 적응시킨 현역 모델, val mAP50 0.931)

## 6. 버전 히스토리 & 성능 추이
| 버전 | 내용 | val mAP50 |
|------|------|-----------|
| v1 | 실사 817 + 합성 2,000 | 0.856 |
| v2 | 1·2차 라벨 검수 반영 재학습 (character P 0.923) | 0.923 |
| **v3 (현재 운영모델)** | **영상 프레임 721장(pseudo-label) 추가 · 영상 도메인 적응** | **0.931** |
| v4 (데이터 준비됨) | 영상 프레임 검수(NPC 오탐 107 제거) 반영 · 학습 대기 | — |

> **v2→v3 트레이드오프:** 동영상 추적률 0.840→0.858 · 캐릭터 탐지율 0.885→0.888(↑) ↔ 정적 mAP50-95 0.591→0.573 · Recall 0.858→0.834(↓)

## 7. 불안정한 하드웨어 위의 학습 — “청크 그라인딩”
- **문제:** GPU 풀로드(학습) 시 전원이 순간 차단되어 강제 재부팅(정전형 크래시). 전력제한·클럭고정으로도 막을 수 없었음.
- **해결:** 2~8 epoch 단위로 끊어 학습 후 epoch 경계에서 정지 → 가중치·옵티마이저·EMA·LR 스케줄을 resume 로 복원해 다시 이어붙이는 **청크 그라인딩**으로 80 epoch 완주 성공. 워커 좀비를 트리킬로 정리하고, GPU 메모리가 idle 로 떨어진 뒤 재실행하는 절차 확립.

## 8. 왜 API가 아닌 ‘화면 인식(Vision)’인가?
**블랙박스 테스트의 본질 — “실제 유저와 100% 동일한 환경”**  
내부 API·로그·메모리 훅은 게임이 *내부적으로* 무엇을 하는지 검증합니다. 하지만 **유저가 실제로 보는 것은 모니터에 그려진 최종 픽셀**입니다. 내부 좌표는 정상인데 렌더링 단계에서 캐릭터가 UI에 가려지거나, 이펙트가 캐릭터를 덮거나, 프레임이 늦게 그려지는 “체감형 결함”은 오직 화면을 직접 보는 방식으로만 잡을 수 있습니다. 게임 엔진·안티치트에 손대지 않으므로 **보안이 철저한 라이브 빌드에도 그대로 적용**됩니다.

- 🚀 **향후 확장:**
