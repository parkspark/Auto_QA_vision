# 던파 캐릭터 탐지·추적 프로젝트 — 전체 진행 이력

> 다른 서비스의 AI 에이전트도 사전 맥락 없이 이해할 수 있도록, 용어 정의·파일 경로·문제→원인→해결→결과 구조로 작성한 자기완결적(self-contained) 문서입니다.
> 프로젝트는 두 코드베이스에 걸쳐 진행됐습니다: `dataminiproject/`(1세대, 정적 탐지) → `MINI_DATA_PROJECT/`(2세대, 영상 추적).

---

## Context (공통 전제 — 모든 섹션에 적용)

- **프로젝트**: 2D 픽셀 액션 게임 *던전앤파이터(Dungeon & Fighter, DNF)* 화면에서 객체를 탐지·추적하는 컴퓨터 비전 프로젝트.
- **탐지 대상 2클래스**: `character`(class 0, 모든 플레이어 캐릭터), `user_id`(class 1, 캐릭터 머리 위 닉네임 텍스트).
- **두 코드베이스**: `dataminiproject/`(1세대, YOLOv8 정적 탐지) → `MINI_DATA_PROJECT/`(2세대, YOLO11s 영상 추적). 1세대 클래스명은 `user_character`/`user_id`, 2세대는 `character`/`user_id`.
- **약어**: YOLO = 실시간 객체 탐지 모델 계열. mAP50 = IoU 0.5 기준 평균정밀도(0~1, 높을수록 좋음). imgsz = 학습/추론 입력 해상도(px). pseudo-label = 모델이 자동 생성한 임시 라벨(사람 검수 전).

## 1. 데이터 수집 — 던담(Dundam) 캐릭터 이미지 스크래퍼

- **목표**: 던전앤파이터 외부 랭킹 사이트 Dundam(`https://dundam.xyz`)에서 직업(class/job)별 상위 랭커의 캐릭터 렌더 이미지를 자동 수집해 학습 데이터 소스로 사용.
- **문제**: 대상 사이트가 React 기반 SPA(Single Page Application)라서, HTTP로 받은 초기 HTML에는 랭킹 데이터가 들어있지 않음(JavaScript 실행 후 동적 렌더). 일반적 해결책인 헤드리스 브라우저 자동화(Playwright/Selenium)는 느리고 불안정.
- **원인**: 데이터가 클라이언트 측 JS가 호출하는 내부 API 응답으로만 존재.
- **해결**: 사이트의 JavaScript 번들을 직접 내려받아 분석 → **비공개 JSON API 엔드포인트 `POST https://dundam.xyz/dat/dealerRankingData.jsp`를 역공학으로 발견**. 요청은 쿼리 파라미터(`page`, `type`, `job`, `baseJob` 등) + 빈 바디. 직업 분류 체계(17개 baseJob → 71개 세부 job 매핑)도 번들에 하드코딩된 것을 그대로 추출. 한글·특수문자 파라미터 인코딩이 엔드포인트와 호환됨을 사전 검증.
- **결과**: 브라우저 자동화 없이 **순수 Python `requests` 스크래퍼**로 구현 → 더 빠르고 안정적. 라이브 사이트 대상 end-to-end(페이지 조회 + 이미지 다운로드) 검증 완료.

## 2. 라벨링 → 학습 → 능동학습(Active Learning) 루프

- **목표**: 수집·캡처한 인게임 스크린샷에 캐릭터/닉네임을 라벨링하고 YOLO 탐지 모델을 학습.
- **파이프라인 (1세대, `dataminiproject/`)**:
  1. `step1_convert.py` — LabelMe(=AnyLabeling) 폴리곤 JSON → YOLO 텍스트 포맷 변환(class 0=`user_character`, 1=`user_id`).
  2. `step2_split.py` — 학습:검증 = 80:20 분할(라벨 907장 → train 725 / val 182).
  3. `step3_generate_yaml.py` — 클래스·경로 정의 `data.yaml` 생성.
  4. `step4_train.py` — **YOLOv8n(경량)과 YOLOv8m(고성능)을 동시 학습·비교**(2클래스, 50 epoch, imgsz 640, batch 8).
  5. `step5_inference.py` — 검증셋 추론·바운딩박스 시각화.
- **능동학습 루프 (`step6_auto_label.py`)**: 학습된 모델이 미라벨 이미지(`unlabeled_images/`)를 1차 자동 라벨링하고, **사람이 그리는 대신 검수만** 하도록 결과를 LabelMe JSON으로 저장(`auto_labeled_data/`). 데이터가 쌓일수록 라벨링 비용↓·모델 성능↑.
- **문제**: 오토라벨 461장 중 **316장이 미탐지(빈 라벨)**. 즉 사람 눈엔 명확히 보이는 캐릭터를 모델이 놓침.
- **해결/활용**: 미탐지 케이스를 별도 수집해 다음 학습 라운드의 보강 대상으로 사용(약점 집중 학습).
- **핵심 인사이트**: **더 강한 YOLOv8m 모델이 오히려 화면 중앙 반투명 UI에 가려진 캐릭터를 놓치는** 역전 현상 발견 → "큰 모델 = 항상 우수"가 아니라, 게임 특유의 가림(occlusion) 상황을 정량 진단할 수 있음을 시사.

## 4. 정적 탐지 → 영상 추적으로 진화 (2세대, `MINI_DATA_PROJECT/`)

- **목표 전환**: 모델을 **YOLO11s**로 교체하고, 단일 스크린샷 탐지에서 **동영상 내 "내 캐릭터" 실시간 추적**으로 확장. 추적은 ByteTrack(탐지 결과를 프레임 간 연결해 동일 객체에 ID를 부여하는 멀티오브젝트 트래커) 사용.
- 개발 중 발생한 문제와 해결(각각 독립적):

  - **문제 A — 닉네임이 너무 작아 탐지 누락**: `user_id`가 원본 해상도에서 ~70×24px라 imgsz 640에서는 탐지 한계선에 걸림. → **해결: imgsz=1280으로 학습 해상도 상향**(이후 고정 결정, 640으로 낮추지 않음).

  - **문제 B — Windows에서 학습 크래시**: 학습 시 데이터 캐시 옵션 `cache="ram"`이 멀티프로세싱 워커 생성(spawn) 단계에서 캐시 직렬화 실패("pickle truncated")로 크래시. → **해결: `cache="disk"`(.npy 파일 캐시)로 전환**. 대용량 RAM이 OS 페이지캐시로 디스크 캐시를 흡수해 속도 손해 없음.

  - **문제 C — "내 캐릭터"를 시각만으로 식별 불가**: 내 캐릭터와 파티원의 외형이 동일해 모델이 구분할 수 없음. → **해결: 모델은 모든 캐릭터를 탐지만 하고, "내 캐릭터" 선정은 후처리(A안)가 담당** — 가장 신뢰도 높은 `user_id`(닉네임) 박스 바로 아래에 위치한 `character`를 내 캐릭터로 선택(닉네임은 항상 자기 캐릭터 머리 위에 렌더됨).

  - **문제 D — 추적 ID가 계속 바뀜(진동)**: 병목은 탐지 자체가 아니라 프레임마다 "내 캐릭터 선정 정책"이 흔들리는 것. → **해결: hysteresis(이력현상) 락 도입** — 한 번 잠근 대상을 일정 프레임 유지하고, 짧은 미탐지(coast-grace)는 무시하며, 교체는 충분한 증거(challenge-frames)가 쌓여야 허용. **ID 스위치 276 → 88로 감소**.

  - **문제 E — pseudo-label에 오탐 혼입**: 영상 프레임 자동 라벨에 NPC·소환수가 `character`로 잘못 포함됨. → **해결: O/X/? 클릭 검수 사이클로 오탐 107건 제거**(v4 데이터셋 정제).

- **버전별 성능(val mAP50)**: v1 **0.856**(실사+합성) → v2 **0.923**(검수 반영) → **v3 0.931**(현역, 영상 프레임 추가·도메인 적응) → v4(데이터 준비 완료, 미학습).
- **v2→v3 트레이드오프**: 동영상 추적률 0.840→0.858, 캐릭터 탐지율 0.885→0.888(개선) ↔ 정적 Recall 0.858→0.834(소폭 하락). 원인 = pseudo-label의 가려진 파티원 미탐지(FN)와 모션블러.

## 5. 하드웨어 전원 불안정 우회 — "청크 그라인딩(Chunk Grinding)"

- **문제**: 이 PC는 **GPU를 풀로드(=장시간 학습)하면 전원이 순간 차단되어 강제 재부팅**됨. Windows 이벤트로그상 Kernel-Power 41 + 크래시 덤프 없음 = 전력 공급이 끊기는 "정전형" 장애. GPU 전력제한·클럭고정으로도 막지 못함(근본 원인은 PSU/전원 계통).
- **영향**: 한 번에 80 epoch 같은 긴 학습을 끝까지 돌릴 수 없음.
- **해결**: 학습을 **2~8 epoch짜리 작은 청크로 쪼개 반복 실행**(`resume_train.py`). 각 청크는 epoch 경계(`results.csv` 행 증가 시점)에서 정지 → 다시 resume로 이어붙임. resume가 모델 가중치·옵티마이저 상태·EMA·현재 epoch·학습률(LR) 스케줄을 모두 복원하므로 **최종 모델 품질에 영향 없음**(비용은 재시작 오버헤드뿐). 이 방식으로 80 epoch 완주 성공.
- **운영 주의사항**:
  - 청크 종료 시 반드시 **트리킬**(`taskkill /F /IM python.exe /T`)로 정리 — 단순 프로세스 종료는 워커 좀비가 남아 다음 청크가 행(hang) 걸림.
  - 청크 사이에 GPU 메모리가 idle(<1.5GB)로 떨어진 것을 확인한 뒤 재실행.
  - 추론(추적·평가·웹앱)은 부하가 가벼워 이 크래시 위험이 낮음.
- **근본 해결 예정**: 신규 PSU 도착 후 연속 학습으로 전환.

## 6. 신규 PC로 환경 이전

- **상황**: 프로젝트를 **다른 하드웨어 PC로 가져옴**. `.venv`(가상환경)·`requirements.txt` 없음.
- **문제**:
  - GPU가 **RTX 2070 SUPER(8GB, Turing 아키텍처)** — 원본 PC의 RTX 5090(32GB, Blackwell)과 다름.
  - **Python 3.12 미설치**(현재 3.13/3.14만 존재). 단 `__pycache__`가 cpython-312 → 프로젝트는 3.12 기준으로 작성됨.
  - 프로젝트 문서의 `torch + cu128`은 **5090(Blackwell) 전용**이라 Turing GPU에 부적합.
- **해결**: `uv`(Python 패키지/버전 관리자)로 **Python 3.12.13** 설치 → `.venv` 생성. 이 PC의 드라이버 560.94가 CUDA 12.6까지 지원하므로 **torch 2.12+cu126**(Turing 호환) 설치. CUDA 인식·현역 모델(`runs/df_yolo11s_1280_v3/weights/best.pt`) GPU 추론까지 검증. 없던 `requirements.txt`를 import 분석으로 재구성해 문서화.
- **메모**: 이 하드웨어 차이(5090 아님 → 2070 SUPER)는 향후 학습 설정에 계속 영향을 주므로 별도 기록됨.

## 7. 포트폴리오 이식 (현재 단계)

- 1세대용 `portfolio_app.py`(YOLOv8 n/m·정적 탐지 설명)를 2세대 실제 현황(YOLO11s·imgsz 1280·영상 추적·v1~v4·hysteresis 락·청크 그라인딩)에 맞춰 **재작성**(`MINI_DATA_PROJECT/portfolio_app.py`).
- 실제 존재하는 추적 영상(`outputs/*.mp4`)·비교 이미지(`reports/*_assets/`) 자산만 참조하도록 가드 처리. 데모 실행 안내를 Flask 웹앱 `.bat`(스크린샷 탐지 :5000 / 영상 추적 :5001) 방식으로 교체.

### 프로젝트를 관통하는 두 가지 테마

1. **블랙박스 비전 QA** — 게임 내부 API/메모리에 접근하지 않고 화면 픽셀만으로 캐릭터를 탐지·추적. 안티치트가 있는 라이브 빌드에도 적용 가능한 비간섭적 테스트.
2. **능동학습 + 정량 진단** — 모델이 1차 라벨링하고 사람은 검수만; 추적률·ID 스위치 같은 지표로 실패를 정량화하며 v1→v4로 점진 개선.
